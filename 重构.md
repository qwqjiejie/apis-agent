## P0 — 基础保障

### P0-1: Langfuse 全链路追踪

**现阶段**: 仅 Python logging + HTTP 请求日志中间件，无分布式追踪，无 trace_id 贯穿，调试靠日志 grep

**目标参考**: Multi-Agent-UniversalFramework `app/harness/trace_context.py`
- `TraceContext` 通过 `contextvars` 注入 trace_id，跨 async 边界自动传播
- `TraceIdFilter` 将 trace_id 注入每条日志记录
- Langfuse `CallbackHandler` 覆盖 Agent 调用链 + RAG 检索 + LLM 调用
- `@observe` 装饰器覆盖关键业务节点

**技术方案**:
1. 新增 `src/apis_agent/common/trace_context.py` — contextvars 管理 trace_id/session_id/agent_type
2. 日志注入改用 `logging.setLogRecordFactory()` 方案（替代 Filter），确保所有 logger（含uvicorn）都有 trace_id
3. 新增 `src/apis_agent/common/langfuse_client.py` — Langfuse 客户端初始化 + CallbackHandler + observe 装饰器，未配置时静默 noop
4. 改造 `BaseAgent._build_trace_config()` — 设置 trace contextvars + 返回含 Langfuse callback 的 config dict
5. 改造 `_process_chunks()` — 透传 config 到 `agent.astream_events()`
6. 改造 ChatAgent/DeepResearchAgent/PptBuilderAgent — 调用 `_build_trace_config()` 并传递 config
7. 改造 `rag_service.py` — retrieve/build_context 添加 `@observe` 装饰器

**处理状态**: 已完成

### P0-2: 限流中间件

**现阶段**: 无限流，任何用户/IP 可无限请求

**目标参考**: Multi-Agent-UniversalFramework `app/gateway/rate_limit_middleware.py`
- 滑动窗口算法（Redis sorted set，60s 窗口，6 个时间桶）
- 两级限流：用户级 60/min + IP 级 300/min
- Redis 不可用时自动降级为内存计数
- 豁免路径：health/docs/static

**技术方案**:
1. 新增 `src/apis_agent/api/middleware/rate_limit.py` — RateLimitMiddleware（BaseHTTPMiddleware 子类）
2. 新增限流配置到 `config/settings.py`：`rate_limit_enabled` / `rate_limit_user_per_min` / `rate_limit_ip_per_min` / `rate_limit_window_sec`
3. 改造 `api/main.py` — CORS→RateLimit→log_requests 三层中间件栈
4. Redis sorted set 滑动窗口 + 内存降级（进程内 list + asyncio.Lock）

**处理状态**: 已完成

## P1 — 能力提升

### P1-1: 工具注册表 + 热重载

**现阶段**: 工具硬编码在各 Agent 类中，新增工具需修改 Agent 代码 + 重启服务

**目标参考**: Multi-Agent-UniversalFramework `app/tools/_registry.py` + `app/harness/tool_hot_reloader.py`
- `TOOL_REGISTRY`: `dict[str, BaseTool]` 全局注册表
- `@register_tool` 装饰器：声明即注册，自动发现
- `importlib.import_module()` 扫描 `tools/` 目录下所有 `.py` 文件
- `ToolHotReloader`: `watchfiles` 监听 `tools/` 变更 → 重新 import → 原子替换注册表 → 失败回滚
- 模块卸载 `unregister_module()` 支持文件删除场景

**技术方案**:
1. 新增 `src/apis_agent/tool/registry.py` — TOOL_REGISTRY + register_tool 装饰器 + unregister_module
2. 改造 `tools/__init__.py` — importlib 自动扫描导入所有工具模块，启动日志输出注册数量
3. 改造 4 个工具文件为 @register_tool 装饰器声明（skills_tool 保持动态加载机制不变）
4. 新增 `src/apis_agent/harness/tool_hot_reloader.py` — watchfiles 监听 + 原子替换 + 失败回滚
5. 改造 `chat_agent.py` / `deep_research_agent.py` — 从 TOOL_REGISTRY 获取工具，新增 `invalidate_cached_agents()` 供热重载回调
6. 改造 `api/main.py` — 通过 lifespan 管理 ToolHotReloader 生命周期

**处理状态**: 已完成

### P1-2: RAG 增强（多路召回 + RRF + 精排）

**现阶段**: 单路 Milvus 语义检索 → 去重合并，固定 500 字符分块，无精排

**目标参考**: Multi-Agent-UniversalFramework `app/documents/retrieval.py` RetrievalPipeline 四阶段
1. **AdaptiveQueryRewriter**: LLM 分类查询类型 (FACTUAL/COMPLEX/SPECIFIC) → 自适应生成 0-3 个改写变体
2. **MultiRecall + RRF**: 多路并行检索 + Reciprocal Rank Fusion (k=60) 融合排序
3. **Rerank**: bge-reranker-v2-m3 本地模型精排
4. **DynamicTopK**: 绝对阈值 (0.75) + 相对下降比 (0.25) 自动截断，至少返回 1 条
5. **LLM Relevance Filter**: LLM 判断检索结果是否真正回答问题，过滤高分噪声

**技术方案**:
1. 新增 `src/apis_agent/rag/retrieval_pipeline.py` — RetrievalPipeline 统一检索管线：
   - QueryRewriter: LLM 分类查询类型 (FACTUAL/COMPLEX/SPECIFIC) → 自适应生成 0-3 个改写变体
   - MultiRecall + RRF: 多路并行检索 + Reciprocal Rank Fusion (k=60) 融合排序
   - DynamicTopK: 绝对阈值 + 相对下降比 (0.25) 自动截断
   - LLM Relevance Filter: LLM 判断检索结果是否真正回答问题，过滤高分噪声
2. 改造 `service/rag_service.py` — `build_context` 改为异步，接入 RetrievalPipeline
3. 改造 `agent/base_agent.py` — `_load_messages()` 改为异步，await RAG 管线
4. 精排 (Reranker) 暂跳过：需本地模型或独立 API，作为后续 P2 模型网关的一部分

**处理状态**: 已完成

### P1-3: Agent 间协作（委派模式）

**现阶段**: 3 个独立 Agent 通过显式 URL 路由，无协作，无委派

**目标参考**: Multi-Agent-UniversalFramework `app/agents/` + `app/harness/task_executor.py`
- **Triage DeepAgent** (分流层): 理解意图 → 简单问题直接回答 → 复杂任务委托后台
- **Executor DeepAgent** (执行层): 后台自主规划 → 委派 Specialist → 审批 → 汇报
- **Specialist SubAgents**: 按 `AGENT.md` 声明工具权限，动态发现
- **TaskExecutor**: `asyncio.Task` 驱动后台执行，支持快照恢复 + HITL 中断
- **TaskContextManager**: 三层记忆（Hot/Warm/Cold）

**技术方案**:
1. 新增 `src/apis_agent/agent/triage_agent.py` — Triage DeepAgent（分流 + 直接回答 + 后台任务委派）
2. 新增 `src/apis_agent/agent/executor_agent.py` — Executor DeepAgent（自主规划 + Specialist 委派 + 审批 + 汇报）
3. 新增 `src/apis_agent/harness/task_executor.py` — 后台任务生命周期管理（提交→执行→中断→恢复）
4. 新增 `src/apis_agent/harness/task_context.py` — 任务上下文 + 三层记忆
5. 新增 `src/apis_agent/agent/specialist/` 目录 — Specialist 子 Agent，按 `AGENT.md` 声明式定义
6. 新增 `src/apis_agent/harness/subagent_discovery.py` — Specialist 自动发现（读取 AGENT.md）
7. 改造 `api/routes/agent.py` — 新增单一入口 `/chat` + `/task/*` 端点

**处理状态**: 待开始

## P2 — 完善增强

### P2-1: 模型网关（健康探活 + 熔断 + 降级）

**现阶段**: 静态 LLM 配置，无健康检查，无故障切换，无熔断

**目标参考**: Multi-Agent-UniversalFramework `app/gateway/model_gateway.py` + `circuit_breaker.py` + `gateway_middleware.py`
- `ModelGateway`: 模型注册表 + 角色路由 (CHAT/FALLBACK_CHAT/RETRIEVAL_LLM/RETRIEVAL_REWRITER)
- `CircuitBreaker`: CLOSED→OPEN（连续 5 次失败）→HALF_OPEN（30s 冷却）→CLOSED（探活成功）
- `HealthProbe`: 后台 30s 间隔对所有模型发送 ainvoke 探活
- `GatewayMiddleware`: LangChain AgentMiddleware，健康感知路由 + 失败回写
- 运行时热切换：Admin API 零停机替换模型

**技术方案**:
1. 新增 `src/apis_agent/gateway/` 包：
   - `model_gateway.py` — ModelGateway 单例（注册表 + 角色路由 + 健康存储）
   - `circuit_breaker.py` — 三态熔断器（CLOSED/OPEN/HALF_OPEN + asyncio.Lock）
   - `health_probe.py` — 后台定期探活任务
   - `gateway_middleware.py` — DeepAgent 中间件（健康感知路由 + 失败回写）
   - `types.py` — CircuitState / ModelRole / ModelSpec / HealthRecord
2. 改造 `config/settings.py` — 模型注册表配置（模型名、提供商、角色、接口类型）
3. 改造 `common/llm.py` — ChatOpenAIWithReasoning 适配网关接口
4. 新增 `api/routes/admin.py` — 管理 API（模型列表、热切换、熔断器状态）

**处理状态**: 待开始

### P2-2: 文档处理状态机 + 去重

**现阶段**: 上传→MIME 校验→MinIO/本地→解析→向量化，单一路径无状态机，无去重，无进度推送

**目标参考**: Multi-Agent-UniversalFramework `app/documents/index_manager.py` + `document_processor.py`
- 9 状态流转: `UPLOADED→STORING→QUEUING→DOWNLOADING→LOADING→CLEANING→SPLITTING→INDEXING→INDEXED(FAILED)`
- SHA-256 哈希去重（4 场景：相同文件/重命名/策略变更/替换）
- 幂等 node_id = `file_hash_content_hash`
- SSE 实时推送处理进度（DocumentEventBus）
- 清洗管线（正则去除 PDF 标记/控制字符/CJK 间距规范化）
- 分块策略工厂（按文件类型自动选择：Markdown/Semantic/SentenceWindow/HTML）
- 知识图谱提取与索引并行

**技术方案**:
1. 新增 `src/apis_agent/document/` 包：
   - `document_status.py` — DocumentStatus 枚举（9 状态）
   - `document_processor.py` — Load→Clean→Parse→Split→Metadata 注入管线
   - `index_manager.py` — IndexManager 状态机编排器
   - `node_parser_factory.py` — 按文件类型选择分块策略
   - `document_event_bus.py` — SSE 文档处理进度推送
   - `datacleaning.py` — 文本清洗（正则管线）
2. 改造 `file_service.py` — 上传时增加 SHA-256 去重 + 状态机触发
3. 改造 `api/routes/file.py` — 新增文档处理进度 SSE 端点
4. 改造 `vector_store.py` — 幂等写入（node_id 去重）

**处理状态**: 待开始
