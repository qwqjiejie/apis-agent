# APIs Agent 架构优化实施计划

> 状态：实施中  
> 基线日期：2026-07-17  
> 架构策略：模块化单体优先，按边界渐进重构，不在领域边界稳定前拆分微服务

## 1. 优化目标

当前项目已经具备多 Agent 编排、后台任务、模型网关、RAG、长期记忆、评估和多种基础设施适配能力。本轮优化不重写业务能力，重点解决以下问题：

1. 建立唯一的应用组合根，消除运行时对象由模块级单例和 `app.state` 分别持有造成的状态分裂。
2. 按业务能力拆分 API 和应用服务，降低 `agent.py` 等大型文件的职责密度。
3. 明确领域层、Agent 运行时层、基础设施层之间的依赖方向。
4. 将数据库变更、部署、测试和架构决策纳入可审计的工程流程。
5. 保持现有 `/api/v1` 接口和核心运行流程兼容，分阶段迁移而不是一次性改目录。

## 2. 当前差距

| 维度 | 当前状态 | 目标状态 | 风险 |
| --- | --- | --- | --- |
| 运行时装配 | 生命周期和多个模块级单例共同持有服务 | `bootstrap` 是唯一组合根，业务代码显式取依赖 | 同一服务可能存在多份状态 |
| 模型网关 | 启动时创建实例，部分代码使用模块级实例 | Agent、RAG、管理接口共享同一实例 | 管理接口看到的状态与真实调用不一致 |
| API 边界 | `agent.py` 混合聊天、任务、网关、反馈、PPT | 按 chat/tasks/admin/feedback 分路由 | 修改冲突和回归范围过大 |
| 领域组织 | document、service、rag、storage 分散协作 | 按 chat/tasks/documents/identity/skills 聚合应用能力 | 跨目录跳转多，所有权不清晰 |
| 基础设施 | `storage` 与 `stores` 并存，存在反向依赖 | 统一 infrastructure 适配器，端口由上层定义 | 难以替换和隔离测试 |
| 初始化时机 | 文件服务、向量库、工具注册存在导入期副作用 | 所有连接和发现行为由生命周期显式启动 | 导入测试可能访问外部服务 |
| 数据库治理 | 无迁移目录和版本基线 | Alembic 管理 schema 变更 | 环境结构容易漂移 |
| 交付治理 | 缺少锁文件、CI、容器和部署基线 | 可复现依赖、分层测试、自动检查 | 本地可用但交付不可复现 |

## 3. 目标架构

目标不是立即移动全部文件，而是先建立边界，再逐步迁移到以下结构：

```text
apis-agent/
├── src/apis_agent/
│   ├── bootstrap/              # 应用工厂、生命周期、依赖装配
│   ├── api/                    # HTTP/SSE 适配层
│   ├── modules/
│   │   ├── chat/               # 对话用例与会话边界
│   │   ├── tasks/              # 后台任务、审批、恢复
│   │   ├── documents/          # 上传、解析、索引、检索
│   │   ├── identity/           # 认证、用户隔离
│   │   └── skills/             # Skill 生命周期
│   ├── agent_runtime/
│   │   ├── agents/             # Triage / Executor
│   │   ├── subagents/          # Specialist 定义与发现
│   │   ├── tools/              # Agent 工具适配器
│   │   ├── prompts/            # Prompt 资产
│   │   ├── memory/             # 短期/长期记忆
│   │   └── evaluation/         # 在线/离线评估
│   ├── infrastructure/
│   │   ├── postgres/
│   │   ├── redis/
│   │   ├── milvus/
│   │   ├── minio/
│   │   ├── neo4j/
│   │   └── llm_gateway/
│   └── shared/                 # 配置、错误、日志、追踪
├── tests/
│   ├── unit/
│   ├── integration/
│   ├── contract/
│   └── e2e/
├── migrations/
├── deploy/
└── docs/adr/
```

迁移期间继续使用 `app/` 包名。只有当模块边界和导入方向稳定后，再单独执行 `src/` 布局迁移，避免目录移动与行为重构混在同一批变更中。

## 4. 依赖规则

```text
API / CLI
   ↓
Application Use Cases
   ↓
Domain + Agent Runtime Ports
   ↓
Infrastructure Adapters
```

- `bootstrap` 可以引用所有层，但业务模块不得反向引用 `bootstrap`。
- API 只负责协议转换、认证和调用用例，不直接创建数据库、模型或存储客户端。
- 领域与 Agent 运行时通过端口依赖基础设施，不从 `service` 反向导入实现。
- 基础设施连接、后台扫描器和热加载器只能在生命周期中启动和关闭。
- 运行时可变对象必须只有一个所有者；兼容别名必须标注迁移期限。

## 5. 分阶段实施

### 阶段 1：建立组合根与唯一运行时容器（P0）

状态：**已完成**

- [x] 新增 `app/bootstrap` 和 `ApplicationContainer`。
- [x] 由 FastAPI lifespan 创建并注册唯一容器。
- [x] Agent、RAG、网关管理接口共享同一 `ModelGateway`。
- [x] Agent API 从容器读取 Agent、任务执行器和长期记忆。
- [x] 工具和 Executor 包装器停止导入模块级 `task_executor`。
- [x] DeadLetter、EventBus、SemanticMemory 改为构造器注入并移除兼容单例。
- [x] FileService 和 VectorStore 移除导入期连接行为。

验收标准：管理接口返回的活跃模型、健康统计和熔断状态与 Agent 实际调用实例一致；重复创建测试应用不会复用上一实例。

### 阶段 2：拆分 API 路由与应用用例（P0）

状态：**已完成**

- [x] 将 `agent.py` 拆为 `chat_routes.py`、`task_routes.py`、`gateway_routes.py`、`feedback_routes.py` 和 `artifact_routes.py`。
- [x] 把会话保存、标题生成、任务查询等逻辑下沉到应用服务。

验收标准：路由函数只包含请求校验、权限、用例调用和响应映射；单个路由模块不超过约 250 行。

## 6. 实施顺序与变更纪律

1. 每个阶段先加保护性测试，再切换依赖或移动代码。
2. 每批变更只解决一个边界问题，保持 API 和存储格式兼容。
3. 兼容层必须在文档中登记，并在后续阶段明确删除任务。
4. 外部基础设施不可用时，单元测试仍应能导入并运行；集成测试单独标记。
5. 目录迁移不与数据库迁移、接口变更同时进行。
