# Dodo-Agent vs TK-MultiAgent - 项目架构对比分析

## 总览

| 维度 | Dodo-Agent (Python) | TK-MultiAgent |
|------|---------------------|---------------|
| **定位** | Java Spring AI 迁移项目，轻量级多 Agent 平台 | 企业级 Multi-Agent 协作系统，生产环境就绪 |
| **代码规模** | ~50 个 Python 文件，~5000 行 | ~80 个 Python 文件，~15000 行 |
| **复杂度** | 中等 — 单一进程，可选基础设施 | 高 — 多 Agent 编排 + 模型网关 + 任务引擎 + 热重载 |
| **Python 版本** | 3.11+ | 3.13.11 |
| **包管理器** | pip + pyproject.toml | uv + pyproject.toml (257 依赖) |
| **部署就绪度** | 开发/内网环境 | 生产环境（多租户 + 限流 + 熔断 + 追踪） |

---

## 1. Agent 编排模型

### Dodo-Agent — 单层 BaseAgent 继承

```
BaseAgent (抽象基类)
  ├── ChatAgent        (聊天 / 文件问答 / 技能助手)
  ├── DeepResearchAgent (Plan-Execute-Critique)
  └── PptBuilderAgent   (状态机 PPT 生成)
```

- **模式**: 一个 Agent 类型对应一个类，继承 `BaseAgent` 共享锁/历史/压缩/清理
- **工具**: 每个 Agent 通过 LangChain `create_react_agent` 绑定固定的工具集
- **路由**: URL 路径决定 Agent 类型 (`/agent/chat/stream` vs `/agent/deep/stream`)
- **并发控制**: `asyncio.Event` 本地 + Redis Pub/Sub 跨实例停止信号
- **断点续传**: PPT Agent 的状态机每步持久化到 MySQL，中断后从中断状态恢复

**优点**: 简单直接，每个 Agent 独立可控，易于理解和维护
**局限**: 无 Agent 间协作，无动态工具发现，无任务委派

### TK-MultiAgent — 两层 DeepAgent 编排

```
Triage DeepAgent (分流层)
  │  理解意图 → 直接回答 / 委托后台任务
  │  工具: 全部注册工具
  ▼
Executor DeepAgent (执行层)
  │  自主规划 → 委派专家 → 审批 → 汇报
  │  工具: request_approval + read_journal
  ▼
Specialist SubAgents (8+ 领域专家)
    动态发现，按 AGENT.md 声明工具权限
```

- **模式**: Triage 分流 → Executor 后台执行 → Specialist 领域执行，LLM 自主规划/委派
- **工具**: `TOOL_REGISTRY` 全局注册，`@register_tool` 装饰器自动发现，按 Agent 角色分层授权
- **路由**: 单一入口 `POST /chat`，LLM 内部判断路由
- **并发控制**: `asyncio.Task` 生命周期管理 + `TaskExecutor` 后台引擎 + `TaskSnapshot` 恢复
- **中断/恢复**: LangGraph `interrupt` + HITL 审批挂起/恢复

**优点**: LLM 自路由，Agent 间可协作委派，工具热重载，专家动态发现
**局限**: 复杂度高，调试困难，依赖 LangGraph 状态管理

### 对比结论

| 特性 | Dodo-Agent | TK-MultiAgent |
|------|-----------|---------------|
| Agent 数量 | 3 个固定类 | 2 个固定 + N 个动态发现 |
| 路由方式 | 显式 URL 路由 | LLM 自路由 |
| Agent 协作 | 无 | Triage → Executor → Specialist 委派链 |
| 工具管理 | 硬编码在 Agent 类中 | 全局注册表 + 装饰器 + 热重载 |
| 断点续传 | PPT 状态机持久化 | LangGraph 状态 + TaskSnapshot 恢复 |
| 学习曲线 | 低 | 高 |

## 2. API 设计

### Dodo-Agent

- **风格**: 全 POST + JSON body，统一 `{code, data, message}` 响应
- **版本**: `/api/v1` 前缀
- **认证**: 无（内网部署）
- **SSE**: 6 个流端点，`sse-starlette` 库
- **端点**: 17 个（agent: 8, session: 4, file: 6）
- **分页**: `pageSize` 上限 100，Pydantic 模型校验
- **文件上传**: multipart/form-data，MIME 类型校验

### TK-MultiAgent

- **风格**: 混合 REST（GET/POST/PUT/DELETE/PATCH），无统一响应格式
- **版本**: 无版本前缀
- **认证**: JWT + 会话所有权校验
- **SSE**: 3 类流端点（chat/task/doc-events），SSE + asyncio.Queue 多路复用
- **端点**: 26 个（auth: 4, chat: 3, session: 5, task: 6, document: 5, admin: 3）
- **分页**: 手动实现，无上限约束
- **文件上传**: multipart/form-data，SHA-256 哈希去重

### 对比结论

Dodo-Agent 的 API 设计更加统一和规范（全 POST + 版本号 + Pydantic 校验 + 分页上限），但缺少认证机制。TK-MultiAgent 功能更全面（任务系统、管理 API、文档 SSE），但 REST 风格不统一且缺少版本控制。

## 3. 中间件体系

### Dodo-Agent

```
FastAPI 中间件:
  ├── CORSMiddleware (allow_origins=["*"])
  └── 请求日志中间件 (URL + 方法)

全局异常处理器:
  ├── ValidationError → 400
  ├── InfrastructureError → 503
  ├── DodoAgentError → 自定义
  └── Exception → 500 (隐藏细节)
```

### TK-MultiAgent

```
FastAPI 中间件:
  ├── CORSMiddleware
  └── RateLimitMiddleware (滑动窗口, Redis + 内存降级)

Agent 中间件管线 (per model call):
  ├── ModelCallLimitMiddleware (100/15 thread/run)
  ├── GatewayMiddleware (健康感知路由)
  ├── ModelRetryMiddleware (1 次快速重试)
  ├── ToolCallLimitMiddleware (50/50 global)
  ├── ToolCallLimitMiddleware (web_search: 20/5)
  └── ToolRetryMiddleware (3 次指数退避)

DeepAgents 框架中间件:
  ├── TodoListMiddleware
  ├── FilesystemMiddleware
  ├── SubAgentMiddleware
  ├── SummarizationMiddleware
  ├── MemoryMiddleware
  └── HumanInTheLoopMiddleware
```

### 对比结论

Dodo-Agent 中间件简洁实用，重点在异常处理。TK-MultiAgent 有三层中间件体系（HTTP → Agent → DeepAgent），生产级关注点全覆盖（限流/熔断/重试/降级）。Dodo-Agent 缺少限流和重试机制。

## 4. RAG / 检索能力

### Dodo-Agent

```
查询压缩重写 → 多查询扩展 → Milvus 语义检索 → 去重合并
```

- **向量库**: Milvus，可选（不可用则跳过）
- **Embedding**: `text-embedding-v4`，通过 API 调用
- **分块**: 固定 500 字符 + 50 字符重叠
- **检索模式**: 简单语义检索，单路召回
- **文件支持**: PDF/DOCX/TXT/图片（多模态 OCR）

### TK-MultiAgent

```
Step1: QueryRewriter (LLM 分类 FACTUAL/COMPLEX/SPECIFIC → 自适应扩写)
Step2: MultiRecall + RRF (多路并行检索 + 倒数排名融合, k=60)
Step3: Rerank (bge-reranker-v2-m3 精排)
Step3.5: LLM Relevance Filter (过滤高分噪声)
Step4: DynamicTopK (按分数分布自动裁剪)
  +
Graph-Enhanced: Neo4j 知识图谱并行检索 (2 跳遍历)
```

- **向量库**: Milvus RBAC 多租户，必需
- **Embedding**: Qwen3-Embedding-0.6B 本地模型 + API fallback
- **Reranker**: bge-reranker-v2-m3 本地模型
- **分块**: 策略工厂 — 按文件类型自动选择 (Markdown/Html/Excel/Semantic/SentenceWindow)
- **检索模式**: 多路召回 + RRF 融合 + 精排 + LLM 过滤 + 图增强
- **文件支持**: PDF (MinerU 本地引擎 + PyMuPDF fallback) / Excel / Docx / Markdown / Html

### 对比结论

Dodo-Agent 的 RAG 是基础实现（单路检索 + 去重），适合简单文件问答。TK-MultiAgent 是生产级 RAG 管线（自适应重查 → 多路融合 → 精排 → LLM 过滤 → 图增强），检索质量显著更高，但复杂度也高出一个量级。

## 5. 文档处理管线

### Dodo-Agent

```
上传 → MIME 校验 → 本地存储 / MinIO → 文本解析 → 向量化 (可选)
```

- 单一处理路径，无状态机
- 解析后立即删除本地临时文件
- 无去重机制
- 无处理进度推送

### TK-MultiAgent

```
上传 → SHA-256 去重 → MinIO → DB 记录 → 状态机驱动处理:
  UPLOADED → STORING → QUEUING → DOWNLOADING → LOADING
  → CLEANING → SPLITTING → INDEXING → INDEXED
  (GRAPH_EXTRACTING 与 INDEXING 并行)
  → FAILED (任意步骤)
```

- 状态机驱动，9 个状态
- 幂等去重 (文件哈希 + 内容哈希)
- 清洗管线 (正则清洗: 零宽字符/PDF 标记/CJK 间距)
- 分块策略工厂 (按文件类型)
- SSE 实时推送处理进度
- 知识图谱提取并行执行

### 对比结论

Dodo-Agent 文档处理简洁直接。TK-MultiAgent 有完整的文档生命周期管理（状态机 + 去重 + 清洗 + 进度推送 + 图谱提取），适合大规模文档处理场景。

## 6. 存储与数据访问

### Dodo-Agent

```
MySQL (SQLAlchemy 2.0) → 会话/文件/PPT 实例
  └── 内存 fallback (不可用时自动降级)
MinIO → 文件对象存储
  └── 本地文件 fallback
Milvus → 向量检索
  └── 跳过 (不可用时)
Redis → 分布式锁 + 跨实例停止信号
  └── 本地模式 fallback
```

- **ORM**: SQLAlchemy 2.0 `Mapped` annotation
- **Repository**: `BaseRepository[M]` 泛型基类，手动 session 管理
- **降级策略**: 所有外部存储均有 fallback，核心服务不中断
- **迁移**: 无 Alembic，手动 SQL

### TK-MultiAgent

```
PostgreSQL 16 (双库) → auth_db + conversations_db (LangGraph 状态)
  └── asyncpg + psycopg AsyncConnectionPool
Redis → 缓存 + 限流 + PubSub
  └── 内存 fallback
MinIO → 文档对象存储
Milvus → 向量检索 + RBAC 多租户
  └── 用户级 Collection 隔离
Neo4j → 知识图谱
  └── 非致命依赖
```

- **ORM**: 无 ORM，直接使用 `asyncpg` + `psycopg`
- **Repository**: 无 Repository 模式，Manager 类直接管理连接
- **降级策略**: PostgreSQL/Milvus 必需，Redis/Neo4j 非致命
- **迁移**: 无 Alembic，lifespan 中手动建表
- **加密**: 敏感凭证 AES 加密后存储

### 对比结论

Dodo-Agent 的核心优势是**全降级设计** — 所有外部依赖可选，核心功能不中断。TK-MultiAgent 依赖更重（PostgreSQL/Milvus 必需），但存储能力更强（双库读写分离 + 图数据库 + 多租户 RBAC）。两者都没有数据库迁移工具。

## 7. 安全机制

### Dodo-Agent

| 机制 | 实现 |
|------|------|
| Shell 安全 | 命令分类 (30+ 读/40+ 危险)，危险命令 SSE 确认，120s 超时 |
| 文件上传 | MIME 类型 + 扩展名校验，大小限制 20MB |
| 会话安全 | `secrets.token_urlsafe(24)` 防枚举 |
| 目录遍历 | `_safe_path()` 防护 |
| CORS | `allow_origins=["*"]` |
| 认证 | 无 |

### TK-MultiAgent

| 机制 | 实现 |
|------|------|
| 认证 | JWT + argon2id 密码哈希 + 会话所有权校验 |
| 限流 | 滑动窗口 (用户 60/min, IP 300/min) |
| 管理 API | X-Admin-Token 认证 |
| 凭据保护 | AES 加密存储 + JWT Secret Key 保护 |
| 多租户 | Milvus RBAC 物理隔离 (每用户独立 Collection) |
| 文档去重 | SHA-256 哈希幂等 |
| CORS | `allow_origins=["*"]` |
| Shell 安全 | 无 Shell 工具 |

### 对比结论

Dodo-Agent 在 Shell 安全性上更完善（这是 LLM Agent 特有的攻击面）。TK-MultiAgent 在认证/授权/多租户方面更成熟。两者 CORS 都是全开放。

## 8. 可观测性

### Dodo-Agent

- **日志**: Python `logging` + 自定义格式，HTTP 请求日志中间件
- **追踪**: 无分布式追踪
- **监控**: 无
- **评估**: 无

### TK-MultiAgent

- **日志**: Python `logging` + `TraceIdFilter`（contextvars 注入 trace_id）
- **追踪**: Langfuse 全链路 (`CallbackHandler` + `@observe` 装饰器覆盖 Agent/RAG/KG)
- **监控**: 模型网关健康探活 (30s 间隔) + 三态熔断器状态
- **评估**: deepEval 离线/在线评估 (Agent + RAG)，Langfuse 反馈回写
- **任务日志**: `TaskJournal` 结构化执行日志 + `StatusCallbackHandler` (15 种状态事件)

### 对比结论

差距最大的一项。TK-MultiAgent 有完整的可观测性体系（追踪 → 日志 → 评估 → 反馈闭环）。Dodo-Agent 仅有基础日志，生产环境需要大幅补强。

## 9. HITL (人审审批)

### Dodo-Agent

- Shell 命令确认: SSE `confirm_shell` 事件 → 前端展示 → 用户 yes/no → `POST /agent/shell/confirm`
- 超时机制: 120s 默认拒绝
- 唯一的人机交互点

### TK-MultiAgent

- 审批工具链: Specialist 发起 → Executor 检测标记 → LangGraph interrupt 挂起 → 前端决策 → resume
- P0 安全兜底: 3 轮未处理审批标记 → `ApprovalNotHandledError` → 强制中断
- 审批持久化: Store + DeadLetterQueue 兜底
- 决策学习: `_learn_from_decisions` 提取审批偏好

### 对比结论

Dodo-Agent 的 HITL 仅覆盖 Shell 安全确认，实现简洁有效。TK-MultiAgent 有完整的审批框架（工具 → 中断 → 恢复 → P0 兜底），可扩展到任意审批场景。

## 11. 扩展性与热重载

### Dodo-Agent

- 新增 Agent: 继承 `BaseAgent` → 实现 `run()` → 添加路由
- 新增 Tool: 在 `tool/` 创建文件 → 在 Agent 类中引入
- 热重载: 无，需重启服务

### TK-MultiAgent

- 新增 Specialist: 创建 `subagents/{name}/AGENT.md` → 自动发现
- 新增 Tool: `@register_tool` 装饰器 → `tools/__init__.py` 自动扫描
- 热重载: `watchfiles` 监听文件变更 → 原子替换 → 失败回滚
- 模型热切换: Admin API 运行时替换，零停机

### 对比结论

TK-MultiAgent 的扩展性设计更为先进（声明式注册 + 自动发现 + 热重载）。Dodo-Agent 需要修改代码并重启。

## 12. 配置管理

### Dodo-Agent

```python
# pydantic-settings + @lru_cache 懒加载 + model_validator 启动校验
class Settings(BaseSettings):
    llm_api_key: str = ""
    ...

@lru_cache
def get_settings() -> Settings:
    return Settings()
```

- 类型安全 + IDE 补全
- 启动时校验关键配置 (api_key 非空 + base_url 格式)
- 有降级策略的配置不做强制校验
- 配置项: 40 个

### TK-MultiAgent

```python
# 普通 dict + get_config() 函数
def get_config() -> dict:
    return { ... }
```

- 无类型安全
- 无启动校验
- API Key 从环境变量直接读取
- 配置项: 30+ 个 (含模型注册表、超时、限流参数)

### 对比结论

Dodo-Agent 的配置管理更规范（pydantic-settings + 类型安全 + 启动校验）。TK-MultiAgent 的配置项更丰富（模型注册表、超时参数、限流参数），但缺少类型校验。
