### 2. 错误处理与韧性 (9/15)

**优点：**
- Redis/Milvus/MinIO/MySQL 不可用时均有降级策略，不会因基础设施故障导致整个服务崩溃
- 分布式锁 + TTL 防止同会话并发执行
- `asyncio.CancelledError` 传播路径完整
- `_process_chunks` 通过 `asyncio.Queue` 解耦 agent 执行与取消信号，设计得很聪明

**问题：**
- `broad except Exception` 遍布全项目，任何异常都被无声吞掉。例如 `common/redis.py:34` 捕获所有异常仅打 warning，这意味着配置错误（如连接字符串格式错误）和运行时故障无法区分
- `storage/db.py:23` 的 `new_session()` 在任何异常下都返回 `None`，无法区分连接超时 vs. 配置错误 vs. 驱动未安装
- 文件上传无大小限制 — `file.file.read()` 会将整个文件读入内存
- 无请求频率限制（rate limiting）
- `routes/agent.py:202` 对 LLM 输出使用 `ast.literal_eval`，LLM 产生的任意文本都可能导致 `ValueError`
- query 参数无长度校验，超大 query 可能撑爆 token 计数

**修改建议：**
- 区分"预期可恢复错误"和"意外错误"：对基础设施连接失败抛出明确的自定义异常类型，由上层统一处理降级；其他异常应有完整 traceback 日志
- `new_session()` 应抛出明确的异常或在日志中区分错误原因
- 为文件上传添加大小限制
- 将 `ast.literal_eval` 替换为更安全的 JSON 解析或至少包裹 try/except 并忽略

### 3. 配置管理 (8/10)

**优点：**
- 使用 pydantic-settings 提供类型安全和 .env 加载
- `.env.example` 完整，新人可快速上手
- 配置项覆盖全面（LLM、存储、Agent 参数、压缩参数）

**问题：**
- `settings = Settings()` 在 `config/settings.py:78` 模块级别实例化 — 导入时副作用，如果有模块在配置加载前导入会拿到默认值
- 密码字段使用普通 `str` 而非 `SecretStr`，序列化到日志或错误消息时会泄露
- `_ENV_FILE` 路径通过 `parent.parent.parent.parent` 向上 4 级 — 极其脆弱，移动文件即崩溃
- `llm_api_key` 默认值为空字符串但服务运行时必备 — 启动时无验证，运行时才报错
- 没有配置校验（如 `llm_base_url` 必须是合法 URL）

**修改建议：**
- 将 `LLM_API_KEY`、`TAVILY_API_KEY`、`MYSQL_PASS` 等敏感字段改为 `SecretStr`
- 使用 `model_validator` 添加启动时校验（如 API key 非空、URL 格式合法）
- 将 `_ENV_FILE` 改为相对于项目根目录的固定路径：`Path(__file__).resolve().parents[3] / ".env"` 或通过环境变量 `DODO_ENV_FILE` 指定

### 4. 安全性 (7/15)

**优点：**
- `_safe_path()` 在文件系统工具中有效防止目录遍历攻击
- bash 工具有 30 秒超时和 10000 字符输出限制

**问题（按严重程度排序）：**
- **严重**: `bash_tool.py:27` `shell=True` 且 command 来自 LLM 输出 — LLM 被 jailbreak 后可直接执行任意系统命令。应采用 `shell=False` + 命令列表，或至少添加严格的命令白名单
- **高**: CORS `allow_origins=["*"]` — 生产环境允许任意源跨域请求
- **高**: 所有 API 端点无任何认证/授权 — 任何人都可以执行 agent、读写文件、访问会话历史
- **中**: session_id 可预测（`chat_{timestamp}_{random}`），存在会话枚举风险
- **中**: 文件上传无类型验证（MIME type），仅靠扩展名判断
- **低**: 错误响应中直接暴露 `str(e)` 到客户端，可能泄露内部路径

**修改建议：**
- 将 `shell=True` 改为 `shell=False`，LLM 输出解析为命令 + 参数列表
- 添加 API key 或 JWT 认证中间件
- 生产环境将 CORS 限制为具体的前端域名
- 使用 `secrets.token_urlsafe()` 生成 session_id

### 6. API 设计 (6/10)

**优点：**
- 全 SSE 流式响应，前后端协议规范一致
- SSE 事件类型体系完整：thinking / text / tool_start / tool_end / reference / recommend / error / complete
- REST 语义用于 CRUD 操作（session/file）清晰

**问题：**
- Agent 端点全部使用 GET + query params — 长 prompt 和复杂参数受 URL 长度限制。应改用 POST + JSON body
- 响应格式不统一：session 路由有时用 `ok()`，有时直接构造 `JSONResponse`
- 无 API 版本号（如 `/api/v1/agent/chat`）
- 分页参数无上限，`pageSize=999999` 会直接打到数据库
- 错误消息直接透传 `str(e)` 到客户端
- `fileId` 参数在 chat stream 用 `default=""`，在 file stream 用 `Query(...)`，同一概念不一致

**修改建议：**
- Agent 流式端点改为 POST，参数通过 JSON body 传入
- 所有非流式端点强制使用 `ok()` / `ok_paged()` 统一响应格式
- 添加 `pageSize` 上限校验（如 max=200）
- 错误响应使用安全的消息模板，不直接暴露异常信息

### 7. 数据访问与持久化 (8/10) `[部分优化]`

**优点：**
- SQLAlchemy 2.0 风格（`Mapped` annotation）
- 双存储（MySQL + 内存 fallback）保障基础可用性
- 索引设计合理（session_id、create_time、file_id）

**问题：**
- `BaseRepository` 的 session 管理手动且易错：`_own_session` 标记 + 分散的 `commit()`/`close()` 调用
- `BaseRepository.count()` (line 64) 使用已废弃的 `self._s.query()` API（SQLAlchemy 1.x 风格），在 2.0 中应写为 `select(func.count()).select_from(self.model)`
- `AiSessionRepo.list_distinct_sessions` 的 `total = self.count()` 统计的是所有行数而非去重会话数，分页 total 不准确
- 无 Alembic 数据库迁移 — DDL 变更全靠手动执行 SQL 文件
- `Base` 类定义在 `models/ai_session.py` 中，其他模型文件通过 `from ai_session import Base` 导入 — 循环依赖风险
- ~~`ppt_builder.py` 的 `_save_inst` 手动操作 session~~ → `[已修复]` 新增 `PptInstRepo.update_status()` 方法统一管理持久化

**修改建议：**
- 将 `Base` 声明移至 `storage/models/__init__.py`
- 引入 Alembic 管理 schema 迁移
- 将 `_save_inst` 逻辑重构为 `PptInstRepo.update_status()` 方法
- `count()` 方法改用 SQLAlchemy 2.0 的 `select(func.count())` 写法

### 8. 异步/并发设计 (7/10)

**优点：**
- `asyncio.Event` 做取消信号 — 比 bool 标志更安全
- `asyncio.Semaphore` 控制深度研究并发度
- `asyncio.Queue` 解耦 agent chunk 生产与取消监听 — 避免了 `astream_events` 内部无法响应取消的问题
- Redis Pub/Sub 跨实例停止信号
- `_process_chunks` 的 `asyncio.wait` + FIRST_COMPLETED 模式处理取消信号，设计精巧

**问题：**
- `_running_tasks` 是进程内 dict — 多 worker 部署时，stop 请求可能路由到没有该 task 的 worker 实例
- `asyncio.create_task(_bg_compress())` 是 fire-and-forget — 如果压缩失败，抛出的异常无人捕获
- `build_llm()` 每次调用都创建新的 `ChatOpenAI` 实例 — 无连接复用
- `tavily_search` 工具每次调用创建新的 `TavilyClient`
- ppt_builder 的 `_search_images` 每次搜索也创建新 `TavilyClient`
- 全局可变状态（`_chat_agent`、`_skills_agent`、`_running_tasks`）无任何锁保护

**修改建议：**
- LLM 和 TavilyClient 实例化改为模块级单例或使用连接池
- `_bg_compress()` 的 create_task 添加 `add_done_callback` 做错误日志
- 停止信号在 `_running_tasks` 找不到对应任务时，返回给客户端明确的状态（如"任务可能在其他实例上运行，信号已广播"）
