# 重构2 — 对齐修正与未完成项技术方案

> 背景：`重构.md` 由 deepseek 主导执行并标注为"全部已完成"，但经逐文件核对，当前代码在多处与 `重构.md` 设计、与 `流程对比.md` 目标架构不符。本文档不重复 `重构.md` 已正确落地的部分，只聚焦**偏差修正**与**未闭合的业务闭环**，给出具体修改点与技术方案。

## 三、重构.md 标注"已完成"但实际代码偏差（真实现状）

### 偏差 1：统一入口路由路径不符

| | 重构.md 设计 | 实际代码 |
|---|---|---|
| 路由 | `POST /api/v1/chat` | `POST /api/v1/agent/chat` |
| 位置 | `app/api/routes/chat_routes.py` | `app/api/routes/agent.py` |

`agent.py` 的 `router = APIRouter(prefix="/agent")`，经 main.py `/api/v1` 前缀拼成 `/api/v1/agent/chat`。重构.md 要求合并为 `/api/v1/chat`。

### 偏差 2：SubAgent 目录命名不符

| | 重构.md 设计 | 实际代码 |
|---|---|---|
| 目录 | `app/subagents/*/AGENT.md` | `app/agent/specialist/*/AGENT.md` |

`subagent_discovery.py` 里 `SPECIALIST_DIR = .../agent/specialist`。功能等价但与文档不符，且 specialist 命名（`ppt_specialist`）与重构.md 一致。

### 偏差 3：中间件管线未接入 Agent（最严重）

重构.md P1-1 声称"完整中间件管线已完成"，并设计了 `GatewayMiddleware`、`ModelCallLimitMiddleware`、`ToolRetryMiddleware` 等。**但实际 `agent_factory.py` 的 `create_deep_agent` 调用没有传任何 `middleware` 参数**：

```python
# app/agent/agent_factory.py 实际代码
agent = create_deep_agent(
    model=model,
    tools=tools,
    system_prompt=system_prompt,
    subagents=subagents,
    checkpointer=checkpointer,
    store=store,
)   # ← 没有 middleware=...
```

后果：
- `app/gateway/middleware.py`（GatewayMiddleware）写了但**未使用**
- `app/agent/middleware.py`（wrap_tool_with_retry / ToolCallLimiter / ModelRetryWrapper）是包装器模式，**也没被 agent_factory 调用**
- 模型重试/工具重试/调用次数限制/网关熔断在 Agent 链路里**全部不生效**

### 偏差 4：base_agent.py 未删除

重构.md 明确"`[删除] app/agent/base_agent.py`"。实际 `base_agent.py` 仍存在，且 `TriageAgent(BaseAgent)` 仍继承它（提供锁/历史/压缩/trace_config 等方法）。

### 偏差 5：TriageAgent 封装类被 /chat 绕过

`/chat` 路由直接用 `request.app.state.agent`（启动时 create 的单例 DeepAgent）调用 `astream_events`，**没有走 `TriageAgent.run()` 封装**。`TriageAgent` 只在离线评估里被实例化。

后果（TriageAgent.run 里有、但 /chat 链路丢失的逻辑）：
- 语义长期记忆注入（semantic_memory.search → context injection）
- 完整能力前缀解析（/chat 自己又实现了一遍简化版）
- 在线评估记录（online_eval）
- tools_used / references 结构化保存

### 偏差 6：后台 ExecutorAgent 不复用单例且丢参数（最严重）

`create_background_task` 的 `_execute` 里 `ExecutorAgent(snapshot, plan)` 每次 new；而 `ExecutorAgent.run()` 内部又 `await create_executor_agent(...)` 再 new 一个 executor，且**三个关键参数全没传**：

| 参数 | lifespan 单例传了 | 后台 ExecutorAgent 传了 | 后果 |
|---|---|---|---|
| checkpointer | ✅ | ❌ | 后台任务无对话历史持久化 |
| store | ✅ | ❌ | 无跨任务记忆 |
| interrupt_on | `{"request_approval":True}` | ❌ | **后台 HITL 完全失效**——request_approval 不会真正挂起 |

而 lifespan 里 `task_executor.executor_agent = app.state.executor_agent` 这个注入**形同虚设**——`create_background_task` 根本不读它。

### 偏差 7：认证标"跳过"但实际建了，且 /chat 没用

重构.md P1-3 标注"跳过"，但实际 `app/auth.py`（JWT）+ `auth_routes.py` 已建。矛盾在于：认证模块建了，**但 `/chat` 路由没调用 `get_current_user_id`**，userId 直接取请求体（可伪造）。

### 偏差 8：TaskExecutor 多处未闭合

| 能力 | 重构.md 声称 | 实际代码 |
|---|---|---|
| recover_tasks | 已完成 | 方法存在但**lifespan 未调用** + **有 `UnboundLocalError` bug**（recovered 在赋值前被 `+=`） |
| resume 路由 | 已完成 | `task_executor.resume()` 方法有，但**无 `POST /task/resume` 路由** → HITL 恢复闭环断裂 |
| 快照持久化 | Store (PG) | **仍是内存 TaskStore**，未接 pg_store_manager.store |
| Journal | 结构化日志 | `_write_journal` 只是 `logger.info`，`read_task_journal` 返回"暂无记录" |

### 偏差 9：EventBus 未注入 Redis

`app.state.event_bus = EventBus()` 创建时未传 redis_client，导致只有进程内 handler，跨进程 Pub/Sub 不可用。

### 偏差 10：TaskContextManager 缺失

重构.md P1-2 提到 `task_context.py` 含 `TaskContextManager`。实际 `task_context.py` 只有 TaskSnapshot/JournalEntry/TaskStore，**无 TaskContextManager 类**，无 `app.state.context_manager`。

### 偏差 11：目录结构清理未完成

重构.md 要求删除 `chat_agent.py`、`ppt_builder_agent.py`、`deep_research_agent.py`、`storage/base.py`、`storage/db.py`、`storage/models/`。需逐一核对残留（`storage/db.py`、`storage/models/` 在当前代码仍被 session/file 服务引用，不能简单删，需评估）。

**修改点**：`app/harness/task_executor.py` + `app/api/main.py`

修复 `recover_tasks` 的 `UnboundLocalError`：

```python
async def recover_tasks(self):
    recoverable = task_store.list_by_status(TaskStatus.WAITING_HUMAN)
    executing = task_store.list_by_status(TaskStatus.EXECUTING)
    executing.extend(task_store.list_by_status(TaskStatus.CREATED))
    recovered = 0
    for snapshot in recoverable:
        recovered += 1
        logger.info(f"[TaskExecutor] 恢复挂起任务: {snapshot.task_id}")
    for snapshot in executing:
        snapshot.status = TaskStatus.CANCELLED
        snapshot.error = "服务重启，任务丢失"
        task_store.save(snapshot)
        recovered += 1
    if recovered:
        logger.info(f"[TaskExecutor] 启动恢复: {recovered} 个任务")
    return recovered
```

在 lifespan 末尾（`yield` 之前）调用：


#### C1. 快照持久化接 PG Store

**修改点**：`app/harness/task_context.py` 的 TaskStore

新增 `PgTaskStore` 实现，复用 `pg_store_manager.store`：

```python
class PgTaskStore:
    """基于 LangGraph AsyncPostgresStore 的任务快照持久化。"""
    def __init__(self, store):
        self._store = store
    async def save(self, snapshot: TaskSnapshot):
        await self._store.aput(
            ("tasks",), snapshot.task_id,
            {"snapshot": snapshot.to_dict()},
        )
    async def get(self, task_id) -> TaskSnapshot | None:
        item = await self._store.aget(("tasks",), task_id)
        return TaskSnapshot.from_dict(item.value["snapshot"]) if item else None
    # list_by_status / list_tasks 类似
```

lifespan 中：`task_store` 切换为 PgTaskStore 实例（store 可用时），否则回退内存。注意 TaskExecutor/event_bus 等引用的是模块级单例，需改为从 app.state 取，或提供 `init_task_store(store)` 注入函数。

## 七、给"看不懂代码"的导航

当前代码混乱的根因是 deepseek 同时引入了**两套并行的执行路径**：

```
路径1（/chat 实际走的）：
  /chat → app.state.agent（单例 DeepAgent）→ astream_events → SSE
  绕过了 TriageAgent 封装、中间件、语义记忆、评估

路径2（重构.md 设计的，部分残留）：
  /chat → TriageAgent.run()（类封装）→ 内部 create_triage_agent（每次new）
  → 含语义记忆/评估，但丢 checkpointer 单例
```

**修正的核心原则：统一到"单例 agent + 单一路径"，让 /chat 的 event_generator 成为唯一执行入口，TriageAgent 类要么删要么只用于离线评估，中间件和后台复用都挂在单例上。** 这样代码就不再有两条路径打架。
