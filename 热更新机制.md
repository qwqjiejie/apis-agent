# 热更新机制

## 核心原理

Agent 启动时创建为**单例**（`app.state.agent`），但这只是一个 Python 对象引用。
热更新通过 **更新 TOOL_REGISTRY → 重建新 Agent → 原子替换引用** 实现，过程中：

- **进行中的请求**：继续使用旧 Agent（引用在请求开始时已捕获）
- **新请求**：使用新 Agent（读 `app.state.agent` 拿到的是替换后的引用）
- **旧 Agent**：无引用后自动被 Python GC 回收

关键代码位于 `api/main.py:29-71`。

## 工具热更新

### 触发：`tool/` 目录下任意 .py 文件变更

**第1步**：`ToolHotReloader` 检测到文件变更
```
文件: harness/tool_hot_reloader.py:59  _handle_changes()
→ watchfiles awatch() 监听 tool/ 目录
→ 检测到 .py 文件 add/modify/delete
```

**第2步**：重新导入模块 → TOOL_REGISTRY 自动更新
```
文件: harness/tool_hot_reloader.py:87-90
→ importlib.reload(module)  或  importlib.import_module(module)
→ 模块顶层代码重新执行
→ @register_tool 装饰器重新触发
→ TOOL_REGISTRY[name] = 新工具对象    （覆盖旧工具）
→ _TOOL_SOURCES[name] = module_name
```

**第3步**：回调 `_rebuild_agents(app)`
```
文件: harness/tool_hot_reloader.py:108-113
→ if reloaded and self._on_reload is not None:
→     await self._on_reload()    # 即 api/main.py 中的 _rebuild_agents
```

### TOOL_REGISTRY 的更新时机

```
文件变更
  → importlib.reload()          ← TOOL_REGISTRY 此时更新
  → _rebuild_agents()           ← Agent 此时重建
```

`TOOL_REGISTRY` 是一个**模块级 dict**（`tool/registry.py:3`），在所有模块之间共享。
`importlib.reload()` 重新执行模块代码时，`@register_tool` 装饰器直接写入这个共享 dict。

## Agent 原子重建

### `_do_rebuild(app)` — 完整流程

```
文件: api/main.py:38-71

async def _do_rebuild(app):
    # 1. 重新发现 SubAgent（读最新的 AGENT.md）
    subagents = _build_subagents_from_specialists()          # 行 44

    # 2. 重新构建 prompt
    triage_prompt = build_triage_prompt()                    # 行 45
    executor_prompt = build_executor_prompt()                 # 行 46

    # 3. 从 TOOL_REGISTRY 取最新工具
    executor_tools = [TOOL_REGISTRY[n] for n in ...]         # 行 48

    # 4. 创建新的 Agent 实例
    new_triage = await create_triage_agent(...)              # 行 52-57
    new_executor = await create_executor_agent(...)           # 行 60-63

    # 5. 原子替换
    app.state.agent = new_triage                             # 行 66
    app.state.executor_agent = new_executor                  # 行 67
    app.state.specialist_subagents = subagents               # 行 68

    # 6. 更新 TaskExecutor 引用
    task_executor.executor_agent = new_executor              # 行 70
```

### 原子性保证

```
api/main.py:25-26
_rebuild_lock = asyncio.Lock()    # 串行化重建请求
_REBUILD_COOLDOWN = 3.0           # 防抖：3秒内多次变更合并为一次重建

api/main.py:30-36
async def _rebuild_agents(app):
    async with _rebuild_lock:                    # 拿锁
        if now - _last_rebuild_time < 3.0:       # 防抖检查
            return                               # 3秒内跳过
        await _do_rebuild(app)                   # 执行重建
```

Python 对象引用赋值 (`app.state.agent = new_triage`) 是原子操作——不会出现"读到一半新一半旧"的情况。

### 旧请求不受影响

```python
# routes/agent.py:79
agent = request.app.state.agent    # ← 请求开始时捕获引用

# 此时热更新发生：app.state.agent = new_agent

# 这个请求继续使用旧 agent（agent 变量已经绑定到旧对象）
async for chunk in agent.astream_events(...):
    ...
```

`agent` 是局部变量，指向旧对象。`app.state.agent` 被替换后，旧对象引用计数减 1 —— 但当前请求还在使用它，所以不会被回收。请求结束后，旧对象引用归零，GC 回收。

## 时间线示例

```
t=0   启动：app.state.agent = Agent_v1 (tools: [A, B], subagents: [S1, S2])

t=10  用户请求1到达 → agent = app.state.agent (= Agent_v1)
      请求1开始执行...

t=12  开发者修改 tool/new_tool.py → ToolHotReloader 检测到
      → importlib.reload(new_tool) → TOOL_REGISTRY["new_tool"] = XXX
      → _rebuild_agents() 触发

t=13  _do_rebuild() 执行:
      → 重新扫描 AGENT.md → subagents 可能包含 new_tool 引用
      → create_triage_agent(tools=latest) → Agent_v2 (tools: [A, B, new_tool])
      → app.state.agent = Agent_v2    ← 原子替换

t=14  请求1 继续使用 Agent_v1 执行（不受影响）

t=15  用户请求2到达 → agent = app.state.agent (= Agent_v2)
      请求2 使用新的 Agent_v2（包含 new_tool）

t=16  请求1 完成 → Agent_v1 引用归零 → GC 回收
```

## 与参考项目的对比

| 维度 | 参考项目 | 当前项目 |
|------|---------|---------|
| 防抖 | 3s cooldown + deferred rebuild | 3s cooldown |
| 并发控制 | `asyncio.Lock` | `asyncio.Lock` |
| 原子替换 | `app.state.agent = new` | `app.state.agent = new` |
| SubAgent 热加载 | `SubAgentHotReloader` | `SubAgentHotReloader` |
| 工具热加载 | `ToolHotReloader` + importlib.reload | `ToolHotReloader` + importlib.reload |
| TaskExecutor 更新 | `app.state.task_executor.executor_agent = new` | `task_executor.executor_agent = new` |
| 失败回滚 | 保留旧 agent（new 赋值前不 delete） | 保留旧 agent（new 赋值前不 delete） |

两者机制一致。关键差异：参考项目有 `deferred_rebuild`（当防抖跳过变更时，调度一个延迟重建任务等剩余冷却时间结束后执行）——当前项目直接跳过不做延迟重建，这可能导致防抖窗口内最后一批变更被忽略。
